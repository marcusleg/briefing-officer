# Enable/Disable Feed Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let users pause a feed so it is skipped by automatic refresh while staying visible (greyed, with a pause icon) and still manually refreshable.

**Architecture:** The `Feed.enabled` boolean (already in `schema.prisma`) gates the bulk-refresh queries (`refreshFeeds`, `refreshCategoryFeeds`) while `refreshFeed` stays unguarded as a manual override. An "Enabled" switch in the existing feed form persists the value; the sidebar renders disabled feeds with muted text and a pause icon.

**Tech Stack:** Next.js (App Router, server actions), Prisma + SQLite, Zod, react-hook-form, shadcn/ui (Radix), Tailwind, lucide-react, Playwright (e2e).

> **Testing note:** This repo has no unit-test runner — the established pattern is Playwright e2e plus TypeScript/lint/build checks. Backend and schema tasks are verified with `npx tsc --noEmit` and `npm run lint`; the user-facing switch and sidebar are covered by a Playwright e2e test. Do not introduce a new unit-test framework.

## Task 7: End-to-end test for enable/disable

**Files:**
- Create: `tests/feed-enable-disable.spec.ts`

This follows the existing Playwright pattern (`tests/navigation-sidebar.spec.ts`, `tests/fixtures.ts`), which provides an authenticated `page` via worker storage state. The test creates a feed, disables it via the edit dialog, and asserts the form round-trips the value.

> Run requires the dev server up (`npm run dev`) per `playwright.config.ts`. If the suite has a `webServer` config it starts automatically.

- [ ] **Step 1: Write the test**

Create `tests/feed-enable-disable.spec.ts`:

```ts
import { expect, test } from "./fixtures";

test("feed can be disabled and re-enabled via the edit dialog", async ({
  page,
}) => {
  // Create a feed to operate on.
  await page.goto("/feed");
  await page.getByText("Add Feed").click();
  await page
    .getByPlaceholder("http://example.org/feed.xml")
    .fill("https://www.theverge.com/rss/index.xml");
  await page.getByRole("button", { name: "Create" }).click();

  // Open the newly created feed, then its edit dialog.
  await page.getByText("The Verge").first().click();
  await page.getByRole("button", { name: "Edit Feed" }).click();

  // The Enabled switch starts on; turn it off and save.
  const enabledSwitch = page.getByRole("switch", { name: "Enabled" });
  await expect(enabledSwitch).toHaveAttribute("data-state", "checked");
  await enabledSwitch.click();
  await page.getByRole("button", { name: "Update" }).click();

  // Re-open the edit dialog and confirm the switch is now off.
  await page.getByRole("button", { name: "Edit Feed" }).click();
  await expect(
    page.getByRole("switch", { name: "Enabled" }),
  ).toHaveAttribute("data-state", "unchecked");
});
```

- [ ] **Step 2: Confirm the Switch is accessible by name**

The Radix `Switch` exposes `role="switch"`. For `getByRole("switch", { name: "Enabled" })` to match, the switch must be associated with the "Enabled" label. The shadcn `FormLabel` renders an `htmlFor` tied to the field id via `FormItem` context, which Radix Switch consumes — so the accessible name resolves to "Enabled".

If the name does not resolve during the run, fall back to locating the switch within its `FormItem` row: `page.locator('[data-slot="switch"]')` scoped to the dialog. Adjust the test to use that locator only if the role+name query fails.
